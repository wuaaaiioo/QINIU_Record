 # 3D 开源模型选型调研与分析
 
 > **技术讨论 PR-1**  
 > 主题：3D 开源模型调研与最终选型——选定 Hunyuan3D
 
 ---
 
 ## 1. 背景与目标
 
 项目需要从图片输入到可打印 3D 模型的完整链路，首阶段聚焦于**开源 3D 生成模型的选型**。
 
 选型核心考察维度：
 
 - 生成质量（几何精细度、纹理一致性）
 - 部署可行性（显存、推理速度、硬件兼容性）
 - 社区活跃度（更新频率、预制生态）
 - 打印适配度（输出格式、后处理工作流）
 
 ---
 
 ## 2. 备选方案调研
 
 ### 2.1 Hunyuan3D 2.0（腾讯混元）
 
 | 维度 | 说明 |
 |------|------|
 | 定位 | 工业级一体化 3D 生成，支持 text-to-3D 和 image-to-3D |
 | 能力 | 多视图生成 + 3D 重建一体化 |
 | 输入 | 单张图片 / 文字描述 |
 | 输出 | 带纹理的 3D 网格（OBJ/GLB） |
 | 优势 | 多视图一致性好，端到端管线完整 |
 ![Hunyuan](https://github.com/wuaaaiioo/QINIU_Record/blob/main/%E8%AE%B0%E5%BD%95/images/hunyuan.png)
 ### 2.2 TRELLIS（微软）
 
 | 维度 | 说明 |
 |------|------|
 | 定位 | 基于结构化 Latent 的 3D 生成，强调可编辑性 |
 | 能力 | 3D 生成 + 结构化解耦编辑 |
 | 输入 | 单张图片 / 文字描述 |
 | 输出 | 结构化 3D 表示（可导出网格） |
 | 优势 | 支持编辑和部件级操作 |
  ![Trellis](https://github.com/wuaaaiioo/QINIU_Record/blob/main/%E8%AE%B0%E5%BD%95/images/trellis.jpg)
 ### 2.3 其他候选方案（未选择）
 
 | 方案 | 未选择原因 |
 |------|------------|
 | **TripoSR** | 单图转 3D 速度极快，但缺少多视图生成能力，精细化不足，后期修复成本高 |
 | **Zero123++** | 主要做 novel view synthesis，需搭配额外重建管线，集成成本高，不符合一体化诉求 |

 ## 3. 本地部署与效果对比
 
 ### 3.1 部署环境
 
 - **硬件**：NVIDIA RTX 4090
 - **框架**：PyTorch + Diffusers / 官方推理脚本

 | 输出格式 | 原生 OBJ/GLB | 需额外导出步骤 |

 ## 4. 选型结论：选定 Hunyuan3D
 
 ### 4.1 选择理由
 
 | 维度 | Hunyuan3D | TRELLIS | 结论 |
 |------|-----------|---------|------|
 | 管线完整性 | 一体化 image-to-3D | 需组合多个组件 | **Hunyuan3D 胜出** |
 | 多视图质量 | 原生优化 | 质量依赖输入 | **Hunyuan3D 胜出** |
 | 社区生态 | 有 ComfyUI 节点、LoRA 支持 | 社区生态较新 | **Hunyuan3D 胜出** |
 | 编辑能力 | 一般 | 强 | TRELLIS 占优，但非当前核心需求 |
 
 **核心结论**：Hunyuan3D 在管线完整性、多视图一致性、社区生态方面均优于 TRELLIS，且与后续 ComfyUI 工作流方案天然兼容。选定 Hunyuan3D 作为主力模型。
 
 ### 4.2 遗留问题
 
 部署实测发现，**直接部署单一 Hunyuan3D 模型**的效果虽优于 TRELLIS，但在以下方面仍有不足：
 
 - 复杂结构的几何丢失/空洞
 - 不可见面的纹理模糊
 - 离可打印级别存在差距
 
 这些问题的解决将在 **PR-2（ComfyUI 工作流方案）** 中讨论。

 ## 5. 附录：验收标准参考
 
 | 检查项 | 标准 |
 |--------|------|
 | 几何完整性 | 无缺失面，非流形边 < 5 处 |
 | 纹理映射 | UV 无撕裂，颜色与参考图一致 |
 | 打印成功率 | 切片成功，无需修复网格 |
 | 实物还原度 | 形态与参考图基本一致 |
